# Handoff Notes - 2026-05-11

This document summarizes the work completed in this session so another coding environment can continue without re-discovery.

## Initial situation

The repository started as a standalone Vite frontend for the "3D Politics Manifesto Remixer".

Critical issues in the original version:

- the browser called Anthropic directly
- the app required the visitor to paste an API key client-side
- `dangerouslyAllowBrowser: true` was used
- deployment assumed GitHub Pages
- there was no backend place to enforce quotas, auth, abuse controls, or provider routing

That made it unsuitable for public release on a real website.

## Architecture direction chosen

The target architecture changed during the session.

### Old assumption

- standalone public app on `remixer.3dpolitics.xyz`

### Final target

- WordPress stays the frontend and content system
- a VPS hosts a reusable AI backend
- no dedicated subdomain is required
- the main site should expose a path like `/ai/v1/*`
- the WordPress-facing server reverse-proxies `/ai/v1/*` to the VPS
- LiteLLM runs on the VPS as the provider gateway
- a thin custom app service sits in front of LiteLLM
- the remixer becomes one use case inside a general AI engine

Recommended topology:

```text
Browser
  -> WordPress page on 3dpolitics.xyz
  -> /ai/v1/* on same domain
  -> reverse proxy to VPS app service
  -> LiteLLM
  -> provider APIs
```

## Major decisions made

### 1. Do not expose provider APIs or keys to the browser

All provider calls must be backend-owned.

### 2. Do not let the frontend choose model/provider/prompt

The backend owns:

- provider selection
- model selection
- prompt template
- max output tokens
- rate limits
- quotas
- logging

### 3. Use-case registry instead of page-specific backends

The backend should support many future features through a registry such as:

- `manifesto_remix`
- `homepage_chat`
- future page assistants
- future policy summarizers

### 4. LiteLLM is a building block, not the full public backend

LiteLLM should handle provider abstraction and spend/route concerns.

The custom app service should still handle:

- WordPress JWT verification
- use-case access checks
- per-use-case validation
- prompt composition
- frontend-safe response shaping

## Architectural caveats

### 1. Two backend paths exist in the repo now

There is:

- the earlier standalone prototype backend in [server.mjs](/home/antoine-vergne/3d-politics-remixer/server.mjs)
- the preferred generic AI gateway in [services/ai-gateway/server.mjs](/home/antoine-vergne/3d-politics-remixer/services/ai-gateway/server.mjs)

The long-term path should be the generic AI gateway.

### 2. The frontend remixer app is still standalone

The current React app in [src/App.tsx](/home/antoine-vergne/3d-politics-remixer/src/App.tsx) is still a standalone Vite UI, not yet a WordPress block/plugin integration.

It should be treated as a source UI that may later be:

- embedded as a WordPress widget
- ported into a WordPress block
- or partially reused in a plugin asset bundle

### 3. LiteLLM model aliases are intentionally generic

In [infra/litellm/config.yaml](/home/antoine-vergne/3d-politics-remixer/infra/litellm/config.yaml), aliases such as:

- `manifesto-remix`
- `homepage-chat`

were used deliberately.

The app service should call business-level model aliases, while LiteLLM maps those aliases to real providers/models.
